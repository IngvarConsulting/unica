# Прототип семантической навигации по объектам 1С

## Статус

Это исполняемый прототип модели, а не обещание готового набора команд. Первый
срез публикует граф в `data.navigation` результата существующего
`unica.meta.info` для объекта в Platform XML. Человекочитаемый текст `meta.info`
и публичный MCP-контракт не меняются.

## Граница миграции legacy-команд

Существующие `unica.*.compile`, `*.decompile` и `*.validate` пока остаются
опубликованным legacy transport. Они не являются `semanticActions` этой модели
и этот прототип не объявляет их заменёнными. Удаление или deprecation возможно
только по каждому специализированному действию отдельно, когда одновременно
доказаны:

1. нативная специализированная операция с её собственным атомарным commit;
2. внутренняя валидация до commit без шага `compile`/`decompile` в публичном
   workflow;
3. совместимый тестовый корпус и явно описанная миграция клиентов;
4. период параллельной поддержки и объявленный срок удаления legacy-контракта.

До выполнения этих условий граф — навигационный и design contract, а не
скрытая замена текущих MCP-тулов.

## Цель

Модель должна позволить перейти от объекта метаданных к его составным объектам
и специализированным операциям без знания XML-файлов:

```text
NodeKind + RelationKind + state -> semantic actions
```

Например, `Document.info` возвращает типизированные ссылки на реквизиты,
табличные части, формы, макеты и команды. Ссылка на форму ведёт в семейство
операций формы; элемент формы в дальнейшем будет иметь другой набор операций,
чем форма или реквизит метаданных.

## Идентичность и граф

`ObjectRef` идентифицирует объект логически:

- source scope: имя настроенного source set либо детерминированный opaque scope
  для ad-hoc XML;
- цепочка владельцев (`ownerChain`);
- вид узла (`NodeKind`);
- имя.

Файловый путь, XML-тег, расположение в выгрузке и внутренний parser handle не
являются публичной идентичностью. Они могут использоваться только адаптером для
чтения или записи.

Для объекта, который не принадлежит настроенному source set, адаптер не
подставляет общий `workspace`: он создаёт стабильный opaque scope из
adapter-private нормализованного контекста. Так два одноимённых ad-hoc объекта
не сталкиваются в `ObjectRef`, но публичный граф по-прежнему не раскрывает путь.

В v1 ссылка адресует снимок по имени и owner chain: после rename клиент получает
новую ссылку из следующего `info`. Отдельный opaque `sourceId`, получаемый из
platform/EDT persistent identifier и не содержащий пути, — следующий срез для
долгоживущих ссылок и повторного разрешения после rename.

Граф состоит из `root`, упорядоченных `nodes` и `edges`. У каждого узла и ребра
есть `capabilityState` и `semanticActions`. В первом срезе Platform XML адаптер
строит связи `contains` для объекта и его реквизитов, табличных частей/колонок,
форм, макетов и команд. Модель также различает `references`: такая связь не
означает владение или право удалить целевой узел.

```text
Document ─contains─> TabularSection ─contains─> Attribute
    │
    ├─contains─> Form ─references─> metadata Attribute / Command
    └─contains─> Template
```

Внутренности формы (атрибуты формы, команды формы, параметры и дерево элементов)
уже имеют отдельные виды узлов в онтологии. Их чтение из Form XML — следующий
вертикальный срез, а не причина считать их обычными XML-фрагментами.

## Семантические действия

`semanticActions` — это **моделируемые возможности**, не имена MCP-тулов и не
заявление о полной реализации. Они вычисляются отдельно для узла и для
конкретного ребра:

- `Document`: изменить свойства, добавить реквизит, табличную часть, форму,
  MXL или команду;
- MXL (`TemplateType=SpreadsheetDocument`): `inspect`, `edit_mxl`, `clone`;
  это отдельный профиль, а не generic `Template`;
- форма: изменить свойства, добавить реквизит/команду/элемент формы;
- элемент формы: собственные изменения, дочерний элемент и обработчик;
- `contains(Form, FormElement)`: `move` — мутация именно связи размещения;
- `references(FormElement, Attribute/FormAttribute)`: связать, перевязать или
  отвязать данные;
- `references(FormElement, Command/FormCommand)`: связать, перевязать или
  отвязать команду.

В первом срезе расширенные профили явно определены только для перечисленных
классов. Неизвестный metadata type, неописанный дочерний класс и non-MXL
template получают только `inspect`, пока для них не появится доказанный профиль.
Так `Configuration` не наследует `add_attribute`/`remove`, а `ActiveDocument`
не выдаётся за MXL.

Это не фиксированный CRUD-список. Действия сформулированы в терминах предметной
модели; `move` и `bind_*` принадлежат relation aggregate, а не самому элементу
формы. `borrow` и `override` остаются кандидатами следующего среза CFE и пока
не включены в исполняемый или модельный профиль.

`clone` показывается на исходном узле, но его commit target — owning `contains`
relation: операция создаёт sibling и его регистрацию одной атомарной фиксацией.
Напротив, `create_handler` остаётся node mutation формы: модуль формы — часть
её object aggregate, а не независимый следующий шаг.

`remove` намеренно пока не рекламируется: для него недостаточно общего
`authorable`, так как поддерживаемый объект можно редактировать, но нельзя
удалить. Оно появится только с отдельным action-specific eligibility состоянием
«снят с поддержки».

## Инвариант мутаций

Каждое действие с `atomic_node_mutation` или `atomic_relation_mutation` является
самостоятельной транзакцией своего агрегата:

1. адаптер подготавливает изменение;
2. выполняет нужную валидацию до фиксации;
3. сохраняет только целостное новое состояние;
4. при ошибке не оставляет этот агрегат в частично записанном виде.

Нет общей неявной транзакции для последовательности вызовов. Если второе
действие цепочки не удалось, успешно зафиксированный результат первого остаётся
корректным. Отдельный пользовательский шаг `validate` не является обязательной
частью будущего API: валидация — обязанность конкретной мутации.

`capabilityState` разделяет две разные причины запрета:

- `resolutionState`: backing source найден (`resolved`) или отсутствует
  (`unresolved`);
- `authorability`: объект можно менять (`authorable`) либо он закрыт
  поставщиком (`support_locked`), глобально read-only
  (`configuration_read_only`) или консервативно неясен.

Мутации появляются только когда обе части состояния это позволяют. Поэтому
форма без Form XML и объект на замке получают только read-семантику, хотя во
втором случае исходник физически существует.

`resolved` означает не просто существование файла. Обычный корневой объект
обязан иметь валидное 1С-имя, совпадающее с именем descriptor-файла;
`Configuration` — исключение с каноническим `Configuration.xml` и
пользовательским логическим `Name`. Form/Template descriptor обязан доказать
свой MD-класс и имя регистрации; MXL — ещё и корректный
`Ext/Template.xml` с известной структурой табличного документа. Несовпавший,
битый или двусмысленный источник остаётся видимым для диагностики, но получает
только `inspect`.

Состояние поддержки вычисляется для descriptor самого узла и консервативно
сочетается с родительским состоянием. Нечитаемый или неразбираемый
`ParentConfigurations.bin` не интерпретируется как разрешение на изменение:
такой объект имеет неясную authorability и получает только read-семантику.
Тот же классификатор использует support guard реальных native-мутаций и
человекочитаемый статус: «не удалось прочитать» блокирует правку, а не
превращается в ложное «не на поддержке».

## Форматы исходников

`representation` записывает, каким адаптером построен граф. Этот прототип
поддерживает только `platform_xml`. EDT — самостоятельное представление с иной
моделью хранения и получит отдельный адаптер; Platform XML граф не должен
выдаваться за EDT-граф. Source format по-прежнему выбирается на уровне source
set, а смешанный/неоднозначный source set остаётся ошибкой границы native XML.

## Что доказывает первый срез

- публичный результат может нести типизированный граф отдельно от прежнего
  текста;
- объектная идентичность не привязана к пути в выгрузке;
- вложенность табличной части и её реквизита сохраняется в owner chain;
- тип макета устанавливается только по descriptor evidence, а не по расширению
  файла;
- корневое имя и typed descriptor/content доказываются структурно до выдачи
  специализированной capability;
- state узла и relation управляет тем, какие семантические действия допустимы;
- vendor lock и global read-only не рекламируют mutation capability;
- lock дочерней формы или макета не маскируется editable-состоянием владельца;
- неоднозначный support state и ad-hoc source scope fail closed, не раскрывая
  путь и не создавая коллизий ссылок;
- `dryRun` сохраняет прежний безопасный placeholder и не читает/не пишет
  navigation target;
- регистрации Form/Template проверяются как 1C identifier до обращения к
  файловой системе;
- политика атомарности является частью модели действия.

1. Platform XML adapter для содержимого формы: атрибуты, команды, параметры,
   дерево form items и их `references`.
2. Специализированные мутации каждого класса узлов с внутренней валидацией и

3. EDT adapter, который строит тот же семантический контракт из EDT model API,

4. Переход от модельных capabilities к исполняемым специализированным
   `unica.*` операциям после проверки их транзакционных границ и тестового
   корпуса.
