# PR #182 Form Design Guide Fixes Design

## Цель

Довести PR #182 до состояния, в котором prompt-visible skill `form-patterns`
полно и честно переносит применимые рекомендации `Oxotka/1CDesignGuide`, каждая
техническая рекомендация соответствует фактическому native-контракту
`unica.form.compile`, происхождение адаптированного MIT-материала отражено в
публичном пакете, а CI стабильно классифицирует изменения в PR.

## Исходные ограничения

- Источником истины для DSL являются Rust-код и тесты, затем package metadata и
  `spec/`; prose не может объявлять несуществующую возможность.
- Публичная граница остаётся одним MCP-сервером `unica` и инструментами
  `unica.*`.
- `form-patterns` остаётся MCP-first и не предлагает прямой запуск скриптов.
- Адаптированный источник закрепляется на commit
  `edc05eaf5c191250a184b0e185006bf4b412f7a5`.
- Copyright notice и полный текст MIT-лицензии источника должны поставляться
  внутри plugin, а `ATTRIBUTIONS.md` остаётся вручную написанной страницей.
- Правила, представленные и в reference spec, и в prompt-visible skill, должны
  быть защищены от смыслового расхождения тестом.

## Рассмотренные подходы

### 1. Только исправить prose

Заменить `buttonHint`, переписать `defaultButton`, добавить недостающие
разделы и attribution. Это минимальный diff, но он оставляет противоречие:
документированный `tooltipRepresentation` и оформление кнопок не эмитятся
native-компилятором.

### 2. Удалить технические примеры

Сохранить только общие UX-принципы и не называть DSL-ключи. Это исключает
ложные инструкции, но делает skill хуже применимым и не выполняет цель PR —
помогать проектировать формы через `unica.form.compile`.

### 3. Исправить prose и минимальный runtime-контракт

Добавить в native compiler ровно те уже документированные свойства, которые
нужны новым правилам: `tooltip`, `tooltipRepresentation`, `backColor` и
`font` у кнопок. Остальные неподдерживаемые виды элементов и оформление не
расширять. Этот подход устраняет корень выявленного противоречия без попытки
закрыть весь накопленный parity backlog.

Выбран подход 3.

## Архитектура изменений

### UX-правила и полнота переноса

Канонический текст раздела `UX-правила для элементов и компоновки форм`
хранится в `plugins/unica/references/specs/form-patterns.md` и зеркалируется в
`plugins/unica/skills/form-patterns/SKILL.md`, потому что skill не имеет
инструментов для чтения reference во время выполнения. Автотест сравнивает
эти секции целиком.

Раздел включает:

- исправленные правила элементов формы;
- реальный DSL-пример подсказки через `tooltip` и
  `tooltipRepresentation: "Button"`;
- одну кнопку с `defaultButton: true`;
- отдельное оформление дополнительных равнозначных действий через
  `font: {"bold": true}` и `backColor: "#FFFF00"`, без второго
  `defaultButton`;
- объяснение двойного отрицания у флажка вместо выглядящей опечаткой фразы;
- рекомендацию делать область ссылки достаточно крупной для уверенного клика;
- бинарный переключатель через поддерживаемый `checkBoxType: "switcher"` и
  UX-правило для многозначного тумблера без ложного примера с пока
  неподдерживаемым native-компилятором `radio`;
- правила обычных, сворачиваемых и всплывающих групп;
- правила командной панели и команд формы;
- правила шапки, подвала и условного оформления.

Существующий общий тезис «кнопки снизу» уточняется по контексту: в
полноэкранной форме основное действие располагается сверху слева, в модальной
форме — снизу справа. Это устраняет противоречие с адаптируемым источником.

Уже существующие подробные DSL-примеры сворачиваемых групп и командных панелей
не дублируются: UX-секция формулирует критерии, а advanced-секция остаётся
реализационным примером.

### Native Form DSL

Компилятор эмитит для поддерживаемых полей и кнопок:

- `<ToolTip>` как локализованный текст;
- `<ToolTipRepresentation>` со значением DSL;
- у кнопки `<BackColor>` как verbatim цвет (`#RRGGBB`, `web:*`, `style:*`);
- у кнопки `<Font .../>` из строковой ссылки на стиль или объекта атрибутов.
- у обычной/сворачиваемой/всплывающей группы `<ShowLeftMargin>` из
  документированного `showLeftMargin`, чтобы рекомендация убирать левый отступ
  не расходилась с native runtime.
- у поля ввода явные `<ChoiceButton>`, `<ShowInHeader>`,
  `<HorizontalAlign>` и `<HeaderHorizontalAlign>` из уже документированных
  ключей, используемых правилами выбора и табличных колонок.
- у кнопки `commandName` как глобальную команду без префикса `Form.Command`;
  `command`, `commandName` и `stdCommand` дают ровно один тег с таким же
  порядком приоритета, а у командной панели `commandSource` как источник `Form` или
  `FormCommandPanelGlobalCommands`;
- дочерние кнопки `cmdBar` получают контекстные типы платформы:
  `CommandBarButton` по умолчанию и `CommandBarHyperlink` для
  `type: "hyperlink"`; обычные кнопки вне панели сохраняют собственные типы;
- у обычного tooltip не только строку, но и документированную многоязычную
  `{ru, en}` форму. Атрибут `formatted` для `<ToolTip>` не добавляется:
  официальное руководство 1С 8.3.27 относит форматирование к `Title`
  расширенной подсказки, а активная XML-spec описывает обычный tooltip как
  `multilang` без атрибута.

Порядок тегов повторяет reference compiler: заголовок и tooltip идут до
common flags; оформление кнопки — после layout/button-свойств и до companion
и events. Реализация не добавляет поддержку новых типов элементов и не
снимает существующий parity gap `element-appearance` целиком.

`showLeftMargin` уже входил в активную DSL-spec и reference compiler, но
native Rust его молча терял. Это обнаруженное при ревью противоречие
устраняется в коде и тесте, а не маскируется оговоркой в prompt-visible skill.

То же правило применяется к обнаруженным при финальном review input-свойствам:
активная DSL-spec и reference compiler уже объявляли их поддержанными, поэтому
native emitter обязан сохранять явные значения вместо silent drop.

Финальное end-to-end review обнаружило те же silent-drop для глобальных
команд/источников и многоязычной объектной подсказки. Они также исправляются
в native коде и точных XML-тестах. Обнаруженное при этом противоречие
формулировок DSL-spec исправляется в самой спецификации: `{text, formatted}`
разрешён только там, где XML несёт `Title formatted`, а не для обычных
`Title`/`ToolTip`. `popup` и `buttonGroup` тоже больше не показаны как
исполняемые элементы, потому что native enum их отклоняет; это ограничение
останется явным до появления реализации и регрессионных XML-тестов. UX-текст
отдельно называет пока неавтоматизируемое устранение дублей стандартных команд
при разбиении табличной части.

Многозначный тумблер в upstream выражен радиогруппой с представлением
`Tumbler`, но native Form DSL ещё не поддерживает `radio`. Skill поэтому
сохраняет применимое UX-правило и прямо запрещает подменять многозначный
элемент бинарным `check`; технический пример появится только вместе с
runtime-поддержкой.

### Provenance и лицензия

`plugins/unica/provenance/skill-upstreams.json` получает upstream
`1c-design-guide` с ролью `guidance`, закреплённым commit и entry для
`form-patterns` со статусом `adapted`. В `plugins/unica/ATTRIBUTIONS.md`
добавляется соответствующий marker, автор, baseline и локальная ссылка на
MIT-лицензию. Полный оригинальный текст лицензии сохраняется в
`plugins/unica/third-party/licenses/1c-design-guide/LICENSE`.

Проверка attribution должна ожидать новый upstream и отвергать отсутствие
его секции или лицензии.

### CI merge-base

`actions/checkout@v7` уже получает полный граф (`fetch-depth: 0`). Повторный
`git fetch --depth=1` делает checkout shallow и при продвижении `main` после
создания synthetic PR merge ref отрезает общий base. Исправление удаляет
`--depth=1`, сохраняя существующие `FORCE_FULL`-ветви и triple-dot diff.

Workflow guardrail требует полный checkout, fetch base без shallow depth и
запрещает возврат проблемной команды.

## Проверки

- Rust unit tests сначала фиксируют отсутствие нужных XML-тегов, затем
  подтверждают tooltip и button appearance.
- Python contract test сначала падает на неполном/невалидном UX-разделе,
  затем проверяет синхронность skill/spec, source URL и допустимые DSL-ключи.
- Attribution tests сначала падают из-за нового ожидаемого upstream marker,
  затем подтверждают полный inventory и локальный license path.
- Workflow test сначала падает на `--depth=1`, затем подтверждает полный
  merge-base path.
- Финальный прогон: Rust workspace tests, Python CI suite, provenance,
  attribution, version contract, `git diff --check` и JSON parsing.
