# Issue #89: надёжность workspace-service

Дата: 2026-07-16

## Цель

Закрыть три причины зависаний публичных инструментов Unica в multi-source-set
workspace:

1. несовместимый вызов `bsl-analyzer search --query`;
2. рассинхронизацию workspace root и фактического source root;
3. последовательную обработку запросов без сквозной отмены и гарантированного
   завершения дерева дочерних процессов.

Публичная граница остаётся прежней: один MCP-сервер `unica` и инструменты
`unica.*`.

## Принятые решения

### Общая архитектура

Публичный stdio MCP-сервер становится диспетчером запросов. Он продолжает
читать входной поток, пока `tools/call` выполняются в рабочих задачах, и хранит
реестр `requestId -> CancellationToken`. Уведомление
`notifications/cancelled` отменяет соответствующую задачу, не ожидая её
завершения.

Внутренний workspace-service принимает соединения конкурентно. Запросы
получают `operationId`; протокол поддерживает независимые команды `ping`,
`cancel` и `shutdown`. Control-команды не ставятся в очередь за анализом или
индексной операцией.

Рабочие операции разделены по ответственности:

- RLM-команды выполняются независимо и не удерживают состояние analyzer;
- доступ к тёплой workspace-сессии `bsl-analyzer` сериализован только внутри
  analyzer lane;
- зависшая операция не блокирует listener и control path.

`shutdown` прекращает приём новых рабочих запросов, отменяет активные операции,
завершает дочерние процессы, удаляет `service.json` и останавливает сервис.

### Выбор source root

Единый `SourceRootResolver` используется analyzer-, RLM- и code-navigation
операциями. Если `sourceDir` не передан, resolver выбирает:

1. source set с именем `main`;
2. при его отсутствии — единственный source set типа `configuration`;
3. при нескольких подходящих кандидатах возвращает контролируемую ошибку с
   требованием передать `sourceDir`.

Fallback на workspace root запрещён, если он не был выбран по этим правилам.
Явный `sourceDir` обязан находиться внутри workspace.

`unica.project.map` остаётся read-only и не создаёт скрытого состояния,
зависящего от порядка вызовов. Сохранение выбора означает, что `project.map` и
последующие операции используют один resolver и получают одинаковый effective
source set/source root. Результат `project.map` сообщает этот effective выбор.

Перед вычислением service key и cache key пути приводятся к единому абсолютному
виду. На Windows эквивалентные обычные и extended-length (`\\?\`) пути должны
давать одинаковый ключ.

### Поиск кода и контракт bundled tool

Общий текстовый поиск `bsl-analyzer search --query` удаляется: закреплённый CLI
предоставляет namespace `search`, а не такую операцию произвольного поиска.
`unica.code.search` агрегирует RLM и `git grep`. Успех любого доступного backend
остаётся успехом, а диагностика недоступного или упавшего backend сохраняется.

CI не содержит жёстко заданного номера версии `bsl-analyzer`. Версия, source tag
и checksum принадлежат `plugins/unica/third-party/tools.lock.json` и
provenance-проверкам. Контрактный тест запускается над фактически
собранным/упакованным бинарником из lock-файла и проверяет поддерживаемый CLI.

### Отмена и дочерние процессы

Все долгоживущие или ограниченные таймаутом процессы запускаются через общий
`ManagedChild`. Он владеет cancellation token, reader-задачами stdout/stderr и
платформенным контейнером дерева процессов.

- На Windows процесс помещается в Job Object с семантикой
  `KILL_ON_JOB_CLOSE`.
- На Unix процесс запускается в отдельной process group, и отмена завершает
  группу.

Отмена возникает при `notifications/cancelled`, внутренней команде `cancel`,
таймауте, разрыве соединения либо `shutdown`. После сигнала отмены применяется
ограниченный grace period. Сбор вывода и join reader-задач также ограничены;
неограниченных `wait()`, `join()` и `wait_with_output()` после kill нет.

Если отменена отдельная операция, workspace-service остаётся работоспособным.
Повторный запрос не требует ручного завершения процесса или перезапуска
сервиса.

## Поток данных

1. MCP dispatcher принимает `tools/call`, создаёт token и регистрирует request.
2. Application layer определяет workspace и effective source root.
3. Для workspace-операции manager создаёт `operationId` и вызывает service.
4. Service направляет операцию в RLM- или analyzer lane.
5. Результат или контролируемая ошибка возвращаются ровно один раз.
6. Завершение удаляет request/operation из реестров.
7. Отмена на любом уровне каскадно переключает тот же token и завершает
   принадлежащее операции дерево процессов.

## Ошибки

Внутренние ошибки различаются как минимум по категориям:

- `invalid_source_root`;
- `timeout`;
- `cancelled`;
- `backend_unavailable`;
- `process_failed`.

Они преобразуются в контролируемый MCP result/error. Ни одна ветка ошибки не
может оставлять неограниченное ожидание процесса, pipe reader или worker thread.

## Тестирование

### Unit

- явный `sourceDir`;
- выбор `main`;
- выбор единственной конфигурации;
- неоднозначный source set;
- эквивалентность обычного и `\\?\`-пути;
- отсутствие формирования `bsl-analyzer search --query`;
- категории timeout/cancelled/process failure.

### Протокол

- долгий data-запрос не блокирует `ping`;
- `cancel(operationId)` завершает только указанную операцию;
- `shutdown` отменяет все операции и отклоняет новые;
- завершившаяся или отменённая операция удаляется из реестра;
- сервис принимает новый рабочий запрос после отмены предыдущего.

### Process tree

Платформенный тест запускает fixture-процесс, который создаёт дочерний процесс.
После отмены проверяется завершение обоих PID. Windows-тест проверяет Job Object,
Unix-тест — process group. Таймауты тестов короткие и детерминированные.

### Интеграция

Fixture-workspace содержит:

- `main -> src/cf`;
- `TESTS -> exts/TESTS`.

Параллельные `unica.code.search` и `unica.meta.profile` без `sourceDir` должны
использовать `src/cf`. Отмена одного запроса не мешает завершению остальных,
не оставляет orphan-процессов и не ломает повторный одиночный вызов.

Контрактный CI запускает проверки над bundled tools, полученными согласно
`tools.lock.json`, без дублирования конкретного номера версии в тестовом коде.

## Критерии готовности

- каждый публичный вызов завершается результатом, контролируемой ошибкой,
  отменой или таймаутом;
- `ping` и `shutdown` доступны во время долгой рабочей операции;
- клиентская отмена доходит до workspace-service и дерева процессов;
- `code.search` не вызывает несуществующий CLI;
- multi-source-set workspace использует согласованный effective source root;
- эквивалентные Windows-пути переиспользуют один service/cache key;
- после отмены сервис принимает последующие запросы;
- проходят `cargo fmt --all -- --check`,
  `cargo clippy --package unica-coder --all-targets -- -D warnings`,
  `cargo test --package unica-coder`, Python CI tests и `git diff --check`;
- runtime/acceptance/ADR-документация соответствует конкурентной и
  cancellable-модели.

## Неграницы

- новые публичные MCP-серверы или публичные control tools;
- смена закреплённой версии `bsl-analyzer` только ради этого исправления;
- файловый watcher или пользовательский глобальный daemon;
- параллельное выполнение запросов внутри одной mutable workspace-сессии
  `bsl-analyzer`.
