# Соглашения по метаданным 1С

Прикладные правила именования и заполнения свойств объектов метаданных.
Смысловые проверки (число, необходимость заполнения и значение синонима)
выполняет разработчик. `meta-validate` проверяет детерминированную длину текста
командного интерфейса для языков конфигурации, но не считает пустой синоним
дефектом: перевод может быть не готов, а представление может формировать
платформа.

Опорные стандарты: v8std 550 «Имена объектов метаданных», v8std 474 «Имя,
синоним, комментарий», v8std 478 «Подсказка и проверка заполнения».

Чек-лист `TemplatesNewObject1C`, из которого также адаптированы правила ниже,
подготовлен для «1С:Бухгалтерии предприятия» и предупреждает, что для других
конфигураций состав настроек может отличаться. Unica осознанно принимает
перечисленные правила как общие проектные соглашения Unica; это не требования
платформы. Если прикладной смысл конкретной конфигурации требует отклонения, его
следует зафиксировать в проекте.

## Имя

- Документ — в единственном числе: `ЗаказПокупателя`, `ПеремещениеТоваров`.
- Избегать слов, от удаления которых смысл не меняется («Документ…»).
- Стараться отразить в имени суть процесса, а не название печатной формы
  (`СверкаВзаиморасчетов`, а не `АктСверки`). Имя от печатной формы допустимо
  только когда объект существует ради этой формы (`ТоварноТранспортнаяНакладная`).

## Синоним

- Проверять необходимость заполнения: синоним участвует в формировании
  пользовательского интерфейса, но может намеренно оставаться пустым, если
  представление формирует платформа.
- Осмысленно и лаконично описывать объект.
- Без сокращений, кроме общеупотребительных аббревиатур (НДС, МСФО).
- Требование распространяется и на реквизиты, табличные части, измерения и
  ресурсы.

## Комментарий

- Не заполнять.

## Представления

- **Представление объекта / записи** — в единственном числе (`Заказ покупателя`).
- **Представление списка** — во множественном числе (`Заказы покупателя`). Не
  задавать, если совпадает с синонимом.
- Слова типа «Поступление», «Реализация», «Инвентаризация» использовать без
  изменения и для объекта, и для списка.
- Длина текста в командном интерфейсе — не больше 38 символов, лучше уложиться
  в 30. Правило применяется только к типам с платформенным свойством
  `ListPresentation`: `ExchangePlan`, `FilterCriterion`, `Catalog`, `Document`,
  `DocumentJournal`, `Enum`, `ChartOfCharacteristicTypes`, `ChartOfAccounts`,
  `ChartOfCalculationTypes`, `InformationRegister`, `AccumulationRegister`,
  `AccountingRegister`, `CalculationRegister`, `BusinessProcess`, `Task`.
  Фактическая область проверки — пересечение этого списка с типами, которые
  поддерживает `meta-validate`.
- Для любого объекта конфигурации или расширения `meta-validate` читает
  владеющий `Configuration.xml` и требует точную регистрацию пары
  тип/имя в `Configuration/ChildObjects`.
- Для каждого зарегистрированного языка код берётся только из
  `Languages/<Name>.xml`. Сначала проверяются все непустые значения
  `ListPresentation` с этим кодом; если их нет — все непустые значения
  `Synonym`. Наблюдаемые `v8:lang` и значения без языка не заменяют языковой
  профиль конфигурации.
- Отсутствующий или пустой перевод пропускается. Отсутствие владельца,
  регистрации, списка языков, зарегистрированного языкового файла или
  `LanguageCode` является ошибкой структуры проекта.
- `ExternalReport` и `ExternalDataProcessor` — самостоятельные корневые
  артефакты EPF/ERF. Они не наследуют соседний `Configuration.xml`, а правило
  длины представления списка к ним не применяется.

## Реквизиты, измерения, ресурсы

- **Проверка заполнения** — «Выдавать ошибку», если хотя бы в одном сценарии
  реквизит обязателен. Условные случаи обрабатывать в `ОбработкаПроверкиЗаполнения`
  модуля объекта, а не наоборот.
- **Значение заполнения** — задавать известное значение по умолчанию.
- **Связи параметров выбора** — заполнять, если значение реквизита зависит от
  других реквизитов объекта.
- **Параметры выбора** — заполнять, если значение ограничено заранее известными
  условиями отбора.

## Справочники

- Если код не имеет прикладного смысла — задать длину кода 0. Если код нужен,
  принцип кодирования определяется прикладным смыслом.

## Регистры

- Представление записи — в единственном числе, представление списка — во
  множественном.
- В командный интерфейс выводятся только независимые регистры сведений. Регистры,
  подчинённые регистратору, в интерфейс не выводятся. `meta-validate`
  предупреждает о регистре сведений с `WriteMode=RecorderSubordinate` и
  `UseStandardCommands=true`.
- Регистр, выводимый в командный интерфейс, должен входить минимум в одну
  подсистему с флагом «Включать в командный интерфейс».
